# AGENTS.md

Guidance for AI coding agents working in this repository.

## Repository source of truth

- Treat `main` as the canonical working branch.
- Ignore the many stale/experimental local branches unless the user explicitly names one.
- The branch `codex/comp-harness-offline-hardening` is reference material only. It contains `review-report.md`; use that report as the roadmap/context for follow-up work, but implement changes on top of `main` unless told otherwise.
- Do not overwrite existing uncommitted work. Check `git status --short --branch` before editing.

## Project overview

RedScript (`redscript-mc`) is a TypeScript compiler/toolchain for a typed high-level language that emits vanilla Minecraft datapacks (`.mcfunction` files, no mods required). It includes CLI commands, LSP, linter, formatter, optimizer passes, Minecraft command validation, stdlib modules, and Paper-server integration-test tooling.

## Common commands

```bash
npm run build                     # TypeScript build + postbuild shims + VSCode extension packaging
npm test -- --runInBand           # Full Jest suite; MC integration tests gracefully skip when server is offline
npm test -- -t "name"             # Run tests matching a name
npm test -- path/to/test.ts       # Run one Jest test file
npm run validate-mc               # Minecraft command syntax validation tests
npm run docs:gen                  # Generate stdlib documentation
npm run docs:check                # Check generated docs against ~/projects/redscript-docs
npm run bench                     # Compiler performance benchmarks
```

Notes:
- The Jest config currently emits warnings about `retryTimes` and deprecated `ts-jest` `globals` config; do not treat these warnings as failures unless the task is about test config cleanup.
- Real Paper server integration requires a live server/TestHarnessPlugin at the configured endpoint; offline skips are expected in local runs.

## Code style

- TypeScript strict mode.
- No semicolons.
- 2-space indentation.
- Tests live under `src/__tests__/`.
- RedScript fixtures/examples live under `src/test_programs/`, `src/examples/`, and stdlib/example directories.

## Compiler pipeline

The current pipeline is:

```text
Source → Lexer → Parser/AST → TypeCheck → HIR → MIR → Optimizer → LIR → Emit datapack
```

Important locations:

| Stage/subsystem | Location | Notes |
|---|---|---|
| Lexer/parser | `src/lexer/`, `src/parser/` | Source to tokens/AST. AST types are in `src/ast/types.ts`. |
| Import/preprocess | `src/compile.ts` | Handles import resolution/source preprocessing before the main pipeline. |
| Type checker | `src/typechecker/` | Collects semantic/type diagnostics; also validates decorators. |
| HIR | `src/hir/` | Desugars high-level language constructs. |
| MIR | `src/mir/` | CFG/three-address IR and lowering. |
| Optimizer | `src/optimizer/` | MIR/LIR optimization passes. |
| LIR | `src/lir/` | Near-Minecraft command IR: scoreboards, storage, calls, execute contexts. |
| Emit | `src/emit/` | Renders datapack files, function tags, runtime helpers, source maps. |
| MC validator | `src/mc-validator/` | Partial static validation of generated commands. |
| MC test client | `src/mc-test/client.ts` | HTTP client for Paper TestHarnessPlugin. |
| Test runner | `src/testing/runner.ts` | `@test`/live runner path; must stay aligned with harness endpoints. |
| LSP | `src/lsp/` | Language server and editor-facing diagnostics/features. |
| VSCode extension | `editors/vscode/` | Built output may be regenerated by `npm run build`. |

## Review report roadmap

Follow the priorities from `review-report.md` on branch `codex/comp-harness-offline-hardening`. The key working assumptions are:

1. RedScript already has real compiler architecture; the highest-value work is hardening stage contracts and the validation loop, not rewriting from scratch.
2. The weak point is confidence that generated datapacks really load/run on target Minecraft versions.
3. Current integration tests can skip offline; do not present offline skips as real Minecraft validation.
4. Static validation is partial and historically skipped high-risk macro/function-with-storage cases.
5. `src/emit/compile.ts` is an overloaded orchestration point; refactor by extracting stages, not by doing a big-bang directory move.
6. Keep work incremental and verifiable: add tests/validators before large semantic changes.

## Active roadmap

The current vNext roadmap is `docs/plans/redscript-vnext-roadmap.md`; the compiler/Minecraft hardening detail roadmap remains `docs/plans/compiler-mc-hardening-roadmap.md`. Follow these documents before starting new implementation slices.

Current priority order:

1. Harden the core generated Minecraft command logic first: `execute`, `scoreboard`, `function`, `function ... with storage`, `$...` macro bodies, storage/NBT, and load/tick entrypoints.
2. Use static validators and small golden outputs to pin command/artifact shape.
3. Use a real Paper/TestHarnessPlugin oracle for deterministic core semantics. Offline skips are useful smoke coverage but are not real MC proof.
4. Treat world/visual side-effect commands (`setblock`, `fill`, particles, title, sound, bossbar, entity spawning) as secondary boundary coverage after compiler-semantic commands are hard.
5. Refactor overloaded compiler files only after behavior is pinned. Known responsibility-heavy files include `src/typechecker/index.ts`, `src/emit/index.ts`, `src/cli.ts`, `src/emit/compile.ts`, and `src/ast/types.ts`.
6. Syntax sugar work comes after the core command oracle: add lowering/codegen tests, not just parser tests.

Completed from the report:

- `redscript test --mc-url` now prefers harness `/command` + `/scoreboard` and keeps legacy `/run` + `/score` fallback.
- compile-output `pack.mcmeta` now uses centralized `mcVersionToPackFormat()`.
- `mc-syntax` no longer skips `function ... with storage` and has initial macro/function-with-storage tests.

## Testing expectations

For compiler behavior changes:

- Add or update unit tests at the affected stage.
- Prefer diagnostic snapshots for errors: kind/message/file/line/col.
- For emitter/codegen changes, add golden output coverage or targeted `mc-syntax`/artifact-validator coverage.
- For optimizer changes, verify both generated output shape and behavior/equivalence where possible.
- For Minecraft semantics, distinguish static command validation from actual Paper-server validation.

Useful test layers:

```text
unit: lexer/parser/typechecker/HIR/MIR/LIR/optimizer/emit helpers
static: generated command syntax + datapack artifact validation
golden: stable emitted file trees for small .mcrs programs
offline integration: graceful skips are OK but are not real MC proof
real integration: Paper server + TestHarnessPlugin oracle
```

## Integration-test/harness notes

- Existing Paper integration uses `MCTestClient` and can skip if the server is unavailable.
- The harness should be treated as a server oracle only when it returns structured reload/command/assertion results.
- Avoid flaky wall-clock assumptions. Prefer tick stepping, isolated namespaces/objectives, reset endpoints, and deterministic scoreboard/storage/block/entity assertions.
- A future test descriptor format should let Node compile/install/reload/run/assert without the plugin invoking the compiler.

## Architecture direction

Do not do a wholesale rewrite. Move toward this shape gradually:

```text
Source
  → import/preprocess
  → lexer/parser/surface AST
  → semantic symbols/resources/typecheck
  → HIR
  → MIR CFG + MIR optimizers
  → LIR / typed Command IR
  → artifact graph
  → validators: syntax, references, version compatibility, collisions, budget
  → datapack emit
  → optional Paper server oracle
```

When refactoring `src/emit/compile.ts`, preserve behavior and extract small named stages first:

- `parseSource`
- `resolveImports`
- `mergeModules`
- `runTypecheck`
- `lowerToHIRStage`
- `lowerToMIRStage`
- `optimizeMIRStage`
- `lowerToLIRStage`
- `optimizeLIRStage`
- `collectRuntimeMetadata`
- `emitDatapack`
- `validateArtifacts`

## Language-design caution

The report flags the language surface as broad. Treat advanced/general-language features as maintenance-sensitive unless they are backed by tests:

- interfaces/generics/lambdas/complex struct inheritance
- advanced decorators such as `@retry`, `@memoize`, `@benchmark`, `@profile`
- raw command escape hatches
- version-specific Minecraft features such as function macros

Prefer hardening validation and tests before expanding syntax.
